# ADR-0009: UI 레이아웃 — 사이드바 중심 (yutils 일관성, 가변 폭)

- Status: Accepted
- Date: 2026-05-21
- Deciders: yusik
- Related: [ADR-0007](./0007-content-policy-headline-curation.md), [ADR-0008](./0008-category-taxonomy.md), PLAN §6

## Context (배경)

운영자는 같은 도메인(`jdgrid.com`) 아래에서 `yutils.jdgrid.com` 같은 사이드 프로젝트를 함께 운영 중이다. 같은 운영자·같은 도메인 패밀리이므로 두 사이트 간 **시각·조작 일관성**을 유지하는 것이 자연스럽다 (브랜드 인상 통일, 추후 도메인 간 상호 네비게이션 가능성).

`yutils`의 UI 패턴 요약 (2026-05-21 분석):
- 왼쪽 고정 사이드바 + 메인 콘텐츠 2단 레이아웃.
- 사이드바에 카테고리별로 도구를 묶고, 각 카테고리 옆에 개수 `(N)` 표시.
- 헤더는 로고 + 우상단 액션만 (미니멀).
- 산세리프, 라이트 모드, 여백 충분.

뉴스 사이트 자체 요구:
- 카테고리 8개(ADR-0008)는 헤더 가로 탭에 욱여넣기에 부담스러움 → 사이드바 모델이 자연스러움.
- 트렌딩 키워드 영역이 별도로 필요 (ADR-0005 데이터 모델의 `trends`).
- 야간 사용도 잦으므로 다크 모드 지원이 필요.
- 본문이 헤드라인 + RSS 요약이라 **본문 가독성**이 첫인상을 좌우.

## Decision (결정)

> 운영자 결정 (2026-05-21):
> 1. 사이드바 폭 → **가변 (접기 가능)**
> 2. 카테고리 옆 기사 수 `(N)` → **표시**
> 3. TRENDING 위치 → **사이드바 하단 + 메인 페이지 상단 띠, 둘 다**
> 4. 라이트/다크 → **시스템 기본 follow + 사용자 토글로 스위칭 가능**
> 5. 타이포·톤 → **읽기 편한 톤으로 수정** (yutils 톤보다 가독성을 한 단계 끌어올림)
> 6. 검색 → **v1 보류** (추후 추가 가능성)

### 레이아웃 골격

- **데스크탑**: 왼쪽 사이드바 + 메인 콘텐츠 2단.
- **모바일**: 헤더의 ☰ 햄버거 → 좌측에서 슬라이드 인 드로어로 사이드바 전환.
- **사이드바 폭은 가변**:
  - 펴진 상태 기본 ~256px.
  - 사용자 토글로 접을 수 있음 (아이콘만 표시 ~64px, 또는 완전히 숨김 — 구현 단계 결정).
  - 접힘 상태는 LocalStorage에 저장 (페이지 이동 시에도 유지).

### 사이드바 구성

```
┌──────────────────────┐
│ 📰 CATEGORIES         │
│ ▸ Top         (32)   │  ← 활성 카테고리
│   World       (24)   │
│   Politics    (18)   │
│   Business    (21)   │
│   Tech        (16)   │
│   Science     (12)   │
│   Sports      (19)   │
│   Culture     (14)   │
│ ──────────────────   │
│ 🔥 TRENDING           │
│   🌐 Global          │
│     #G7 summit       │
│     #Trump tariff    │
│     #AI safety bill  │
│   🇰🇷 Korea           │
│     #SK하이닉스       │
│     #총선             │
│ ──────────────────   │
│ About                │
│ GitHub               │
└──────────────────────┘
```

- 카테고리 옆 `(N)` = 오늘자 기사 수 (ADR-0008 단일 분류 기준).
- TRENDING은 칩 리스트 형태로 압축. 클릭 시 `/k/[keyword]`.
- 사이드바는 모든 페이지에 동일하게 노출 (sticky), 현재 페이지의 카테고리만 활성화 표시.

### TRENDING 중복 노출 정책

같은 트렌딩 데이터가 두 군데 보이지만 **컨텍스트가 다르다**:

- **메인 페이지 상단 띠**: 강조용. 큰 글씨, 가로 스크롤, 시각적 임팩트.
- **사이드바 하단**: 빠른 참조용. 작은 칩 리스트, 다른 페이지에서도 항상 보임.

카테고리 페이지나 키워드 페이지에서는 메인 상단 띠가 없고 사이드바 하단의 트렌딩만 보인다.

### 테마

- 기본: **시스템 `prefers-color-scheme` follow**.
- 헤더 우상단의 ☾/☀ 아이콘으로 사용자 토글 (라이트 ↔ 다크 ↔ 시스템 follow 3 상태).
- 선택은 LocalStorage 저장.
- FOUC(첫 페인트 깜빡임) 방지를 위해 `<head>` inline script로 페인트 전 클래스 적용.

### 타이포·가독성

- 산세리프 기조 + **가독성 우선** — yutils 톤보다 본문 폰트 크기·line-height·자간을 한 단계 키운다.
- 본문 기준: `16~17px / line-height 1.55+ / letter-spacing -0.005em` 권장.
- 헤드라인: `font-weight 700~800`, 크기는 카드/페이지 위계에 맞춰.
- 한국어 + 영문 혼용. 폰트 후보: **Pretendard** (한국어·영문 모두 깔끔), 또는 `Inter + Noto Sans KR` 조합. 최종 폰트는 구현 단계에서 결정.

### 검색

- v1 미포함. 카테고리·날짜·트렌딩 키워드 네비게이션으로 충분.
- 데이터 모델은 검색 없는 가정으로 시작하되, 추후 추가 시 호환되도록 `id`/정규화 URL 같은 식별자를 보존한다.
- 도입 시점은 별도 ADR로 결정.

## Consequences (결과)

**긍정**
- yutils와 시각·조작 일관성 → 같은 운영자의 두 사이트를 사용자가 매끄럽게 오감 (브랜드).
- 사이드바 모델이 카테고리 8개 + 트렌딩을 동시에 흡수해 정보 위계가 명확.
- 시스템 follow 다크모드 → 추가 학습 없이 친숙.
- 검색을 빼면서 빌드 산출물·인덱스 단순화 (정적 export 흐름과 일치, ADR-0002).

**부정**
- 가변 사이드바: 접힘 상태 UI(아이콘만)와 펴진 상태 UI 두 벌의 디자인·QA 필요.
- 테마 토글 + LocalStorage: 첫 페인트 시 FOUC 위험 → inline script로 사전 적용 (구현 비용 소).
- TRENDING 둘 다 노출: 중복 정보의 위계 혼란 위험 → 상단 띠=강조 / 사이드바=참조로 컨텍스트 분리 명시.
- yutils 톤보다 가독성 우선으로 키우면 두 사이트 인상이 살짝 달라짐 (의도된 분기).

**중립**
- 폰트는 본 ADR에서 톤만 결정. 구체 폰트(Pretendard vs Inter+Noto Sans KR)는 구현 단계 선택.
- 검색은 v2 이후. 데이터 모델은 호환성 유지.

## Alternatives Considered (대안)

- **헤더 가로 탭**: 카테고리 8개에 트렌딩까지 끼우면 모바일에서 압박. 사이드바 채택.
- **사이드바 고정 폭**: 단순하지만 작은 화면에서 답답함. 가변 채택.
- **다크 only / 라이트 only**: yutils는 라이트 기본이지만 뉴스 사이트는 야간 사용도 많음 → 시스템 follow가 합리적.
- **TRENDING을 별도 페이지로**: 데일리 대시보드 목적과 어긋남, 첫 화면 정보 밀도 ↓.
- **TRENDING 한 군데만**: 운영자가 둘 다 원함, 그리고 강조/참조로 컨텍스트가 다름.
- **검색 v1 포함**: 정적 사이트에 클라이언트 검색 인덱스를 얹는 비용 vs 효용이 v1에선 낮음. 미룸.
- **타이포는 yutils 그대로**: 도구 사이트보다 뉴스는 본문 읽기 시간이 길다 → 가독성을 한 단계 키우는 분기 선택.
