# ADR-0007: 콘텐츠 정책 — 헤드라인 인덱스/큐레이션 (전문 저장·재가공 금지)

- Status: Accepted
- Date: 2026-05-21
- Deciders: yusik
- Related: [ADR-0004](./0004-data-storage-git-json.md), [ADR-0005](./0005-data-sources-rss-google-trends.md)

## Context (배경)

뉴스 어그리게이션 서비스는 매체의 저작권·이용약관과 늘 부딪힌다. 1인 운영 + 무수익 사이트라도 본문 전문을 무단 저장·재게시하면 침해 소지가 분명하다. 반면 **헤드라인 + 짧은 요약 + 원문 링크**는 일반적으로 fair use / 정당한 인용 범위로 받아들여지고, 매체 자신이 RSS에 그 형태로 공개해두는 것이 그 묵시적 동의의 근거가 된다.

저장·표시 가능 영역을 ADR로 박아두면:
- 향후 매체 클레임 시 "우리는 처음부터 이 정책을 가지고 있었다"는 방어 근거가 된다.
- 기여자(또는 AI)가 무심코 본문 prefix를 저장하거나 LLM 요약을 끼워넣는 변경을 막을 수 있다.

## Decision (결정)

> 운영자 결정 (2026-05-21): "초기 버전은 제목, 짧은 요약, 썸네일, 출처, 원문 링크 중심의 뉴스 인덱스/큐레이션 형태로 간다. 요약은 RSS description 그대로 노출, 썸네일은 외부 URL 핫링크."

### 저장·표시 가능 필드 (5개)

1. **`title`** — 매체가 RSS에 노출한 헤드라인.
2. **`summary`** — **RSS `description` 필드를 가공 없이 그대로** 사용. 자체 길이 컷·LLM 재요약 금지.
3. **`imageUrl`** — RSS에 매체가 제공한 대표 이미지 URL을 **외부 핫링크**로 그대로 사용. 자체 다운로드·재호스팅 금지.
4. **`source`** — 매체명 + 매체 사이트 URL.
5. **`url`** — 원문 링크. 클릭 시 매체 페이지로 이동.

부수 메타데이터 (`id`, `publishedAt`, `lang`, `category`)는 표시·검색용 식별자이므로 본 정책의 제약 대상이 아니다.

### 금지 항목

- 본문 전체 저장·표시.
- 본문 일부(첫 N자 prefix, 발췌 인용 등) 저장·표시.
- LLM·룰 기반을 막론한 **자체 요약 생성** — `summary`는 매체가 RSS에 적어둔 그대로만.
- 매체 이미지의 **자체 호스팅 (R2/S3/CDN 캐싱 포함)**.
- 광고·유료 콘텐츠를 무시한 우회 게재.

### 운영 원칙

- RSS에 `description`이 없거나 비어 있으면 `summary`는 빈 값으로 둔다 (대체 요약을 만들지 않음).
- 썸네일이 핫링크 실패(Referer 차단, URL 만료 등)로 깨지면 placeholder로 폴백한다.
- 매체로부터 개별 클레임이 오면 해당 매체 RSS 소스를 `scraper/sources.ts`에서 즉시 제거한다.
- `/about` 페이지에 출처·면책·연락처를 명시한다.

## Consequences (결과)

**긍정**
- 저작권 리스크가 가장 낮은 형태로 운영 가능. 헤드라인 + 매체 제공 요약 + 원문 링크는 fair use/인용 범위.
- "우리가 만든 콘텐츠"가 사실상 없으므로, 우리 사이트의 가치는 **선별·정렬·시각화**에 집중된다 (목적과 일치).
- 이미지 호스팅·요약 파이프라인이 빠지므로 운영 비용·복잡도 추가 0.

**부정**
- RSS `description`이 빈약한 매체(또는 빈 매체)는 카드 UI가 헐겁게 보인다 — 카테고리/매체 가중치로 보완.
- 핫링크 의존: 매체가 Referer 차단하면 썸네일이 깨질 수 있다. 시간이 지나면 만료된 이미지 URL도 늘어남.
- LLM 요약이라는 강력한 차별화 카드를 포기. 추후 매체 합의·라이선스 확보 시점에 재검토 (별도 ADR로 supersede).

**중립**
- 핫링크는 매체 서버에 약간의 부하를 준다. 일 1회 정적 빌드 후 방문자 트래픽 수준이라 무시할 만하지만, 트래픽이 커지면 매체와 마찰 요소가 될 수 있음.
- `data/*.json`의 각 아이템은 매체 표시 단위로 비교적 가벼움 (수 KB) → ADR-0004의 git 저장 가정과 잘 맞는다.

## Alternatives Considered (대안)

- **LLM 요약(Claude/GPT)으로 일관된 1~2줄 생성**: 품질·일관성은 좋지만, (1) 호출 비용, (2) 저작권 회색지대 — LLM 요약도 원문에서 파생된 2차 저작물 성격, (3) 빌드 파이프라인에 외부 API 의존이 들어가 안정성 ↓.
- **200자 자체 컷**: RSS description이 짧은 매체엔 무의미하고, 긴 매체엔 의미를 잘라먹는다. 매체 의도 존중하는 "그대로 노출"이 더 깔끔.
- **본문 prefix 200자 저장**: 매체에 따라 침해 소지. RSS description 범위 안에서만 가는 게 명확한 선.
- **제목만 표시 (summary 없음)**: 가장 보수적이지만 사이트의 정보 밀도가 크게 떨어진다. 운영자가 "짧은 요약, 썸네일 포함"으로 명시함.
- **Cloudflare R2/Images 자체 호스팅**: 핫링크 깨짐 방지엔 좋지만, 우리가 이미지를 "저장·재배포"하는 행위가 추가되므로 저작권 부담이 오히려 커진다. 운영 비용도 추가.
